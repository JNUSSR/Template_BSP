# 03. 驱动、回调与任务启动

## 先看系统级入口

这一节先抓主线，不从单个电机开始看，而是先看：

1. `freertos.c`
2. `tsk_config_and_callback.cpp`
3. `drv_can.c`
4. `drv_uart.c`

这样最容易搞清楚 **系统什么时候初始化、回调怎么分发、任务什么时候开始跑**。

## `Task_Init()` 放在哪里

当前工程里，`Task_Init()` 不是随便找地方调用的。

它被放在 [freertos.c](F:/Code/ROBOCON/2026ROBOCON/Task/BSP+FreeRTOS/Template_Cmake/Core/Src/freertos.c) 的 `MX_FREERTOS_Init()` 里，线程创建之前先执行。

当前实际关系是：

```c
void MX_FREERTOS_Init(void) {
  Task_Init();

  TaskJ4310Handle = osThreadNew(StartTaskJ4310, NULL, &TaskJ4310_attributes);
  Task3508Handle = osThreadNew(StartTask3508, NULL, &Task3508_attributes);
}
```

所以一定要记住：

- **`Task_Init()` 先执行**
- **线程后创建**

## `tsk_config_and_callback.cpp` 是干什么的

这个文件主要用来放两类内容：

- **全局初始化函数**
- **全局回调函数**

它不负责某个设备的具体主循环，而是负责系统级入口。

### `Task_Init()`

当前模板里它做了这些事：

1. `BSP_Init(...)`
2. `CAN_Init(&hcan1, CAN_Motor_Call_Back)`
3. `DJI_M3508_Task_Init()`
4. `J4310_Task_Init()`

也可以理解成它初始化了三大块内容：

- **板级资源**
- **全局通信回调**
- **示例任务所需的设备对象**

### `CAN_Motor_Call_Back(...)`

这是注册给 `CAN_Init()` 的回调函数。

作用是：

- 收到一帧 CAN 报文后
- 根据 `StdId` 判断属于哪个设备
- 再把数据分发给对应设备对象

这类函数的职责是：

- **分发**
- **转发**
- **不做复杂控制**

## `drv_can` 怎么看

### `CAN_Init(...)`

作用：

1. 启动 CAN
2. 打开接收中断
3. 登记接收回调
4. 配置过滤器

**`CAN_Init(...)` 初始化的主要内容是：**

- **启动指定 CAN 外设**
- **打开接收中断**
- **登记接收回调函数**
- **配置默认过滤器**
- **把该 CAN 和管理对象绑定**

当前模板里的典型调用是：

```cpp
CAN_Init(&hcan1, CAN_Motor_Call_Back);
```

含义是：

- 初始化 `CAN1`
- 收到数据以后，交给 `CAN_Motor_Call_Back`

### `TIM_CAN_PeriodElapsedCallback()`

**这是 CAN 的发送部分，不是接收部分。**

它的职责是：

- 把前面已经准备好的发送缓存真正发出去

当前示例代码已经写好的发送对象是：

- **`CAN1`**

如果以后你还需要发 `CAN2`，需要在这个函数里自己补：

```cpp
CAN_Send_Data(&hcan2, ...);
```

所以这里请直接记住：

- **接收：靠中断回调**
- **发送：靠 `TIM_CAN_PeriodElapsedCallback()` 周期下发**

## `drv_uart` 怎么看

### `UART_Init(...)`

作用：

1. 绑定串口管理对象
2. 保存接收回调函数
3. 启动 `DMA + ReceiveToIdle`

**`UART_Init(...)` 初始化的主要内容是：**

- **选择使用哪一路 UART**
- **把 UART 和管理对象绑定**
- **登记接收回调函数**
- **设置接收缓冲区长度**
- **启动 DMA + 空闲中断接收**

### UART 回调链路示例

如果你要把串口收到的数据交给串口绘图对象，可以写成：

```cpp
static Class_Serialplot serialplot;

static void UART_Serialplot_Call_Back(uint8_t *Buffer, uint16_t Length)
{
    (void)Length;
    serialplot.UART_RxCpltCallback(Buffer);
}
```

对应链路是：

```text
UART_Init(&huart2, UART_Serialplot_Call_Back, 64)
  -> HAL_UARTEx_RxEventCallback()
  -> UART2_Manage_Object.Callback_Function(Buffer, Size)
  -> UART_Serialplot_Call_Back(Buffer, Size)
  -> serialplot.UART_RxCpltCallback(Buffer)
```

这和 CAN 的思路是一样的：

- **底层驱动负责收发**
- **系统回调负责转发**
- **设备对象负责解析**
