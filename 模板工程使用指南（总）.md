# 大疆A板 + FreeRTOS + CMake 模板工程使用指南
>
> 目标：看完这份文档后，你应该知道这个模板工程是干什么的、代码应该放在哪一层、关键函数怎么调用、加电机和传感器时应该怎么接入。

---

## 1. 这个工程到底是干什么的？

这个工程是 RoboCon 机器人比赛使用的固件模板，运行平台是 大疆A板（也适用于到C板和dm02），操作系统是 FreeRTOS，构建方式是 CMake，常用开发环境是 VS Code和CLion。

这个模板已经把以下内容准备好了：

- STM32 的底层外设初始化框架
- FreeRTOS 任务调度框架
- `User/` 目录下的分层代码结构
- 电机、CAN、UART、TIM 等常用功能封装

你以后写业务代码，主要工作区域就是：

- `User/`：写固件逻辑

---

## 2. 为什么要分层？

分层的目的很简单：**让每一部分代码职责清楚，便于维护、复用和排错。**

如果不分层，常见问题是：

- 一个文件里同时写通信、解析、控制、调度，代码会很乱
- 新人看不懂，也不知道该从哪里改
- 换一个电机或者换一个传感器时，改动范围会很大
- 出问题时很难判断到底是驱动错了、解析错了还是控制错了

分层以后，每层只负责自己的事：

- `Driver`：负责和硬件接口直接通信，比如 CAN、UART、TIM
- `Device`：负责把原始数据解析成“设备状态”，例如电机角度、转速、温度
- `Algorithm`：负责算法计算，比如 PID、滤波、斜坡规划
- `Module`：负责把设备和算法组合成完整功能，比如底盘控制
- `Task`：负责 FreeRTOS 任务创建、初始化顺序、周期调度、回调分发

简单记：

- 通信问题先看 `Driver`
- 解析问题先看 `Device`
- 控制参数问题先看 `Algorithm`
- 功能逻辑问题先看 `Module`
- 任务调度和初始化问题先看 `Task`

## 3. 命名规范

这个工程用文件名前缀表示代码所在层级。

| 前缀     | 含义        | 说明  |
| ------ | --------- | --- |
| `alg_` | Algorithm | 算法层 |
| `drv_` | Driver    | 驱动层 |
| `dvc_` | Device    | 设备层 |
| `crt_` | Module    | 模块层 |
| `tsk_` | Task      | 任务层 |

例子：

- `alg_pid.cpp`：PID 算法实现
- `drv_can.c`：CAN 驱动
- `dvc_motor.cpp`：大疆电机设备对象
- `crt_wheelchassis.cpp`：底盘模块
- `tsk_config_and_callback.cpp`：任务初始化和回调分发

补充：

1. 任务文件的命名现在统一采用 `_task` 后缀，例如 `motor_dji_task.cpp`、`motor_dm_task.cpp`。

## 4. 根目录说明

### `Core/`

CubeMX 生成的核心代码目录，通常包含：

- `main.c`
- 中断处理文件
- HAL 初始化流程
- `freertos.c`

这里是整个程序的启动入口。可以看，但不建议新手随便大改自动生成部分。

### `Drivers/`

STM32 HAL、CMSIS 等底层库目录。这里主要是官方库代码，平时业务开发一般不用改。

### `Middlewares/`

中间件目录。这个工程里最重要的是 FreeRTOS 源码。一般只使用，不直接修改。

### `User/`

战队自己的核心代码目录。绝大多数业务逻辑、控制逻辑、设备封装、任务代码都放这里。

### `CMakeLists.txt`

工程总构建文件，用来告诉 CMake：

- 工程名称是什么
- 使用哪些语言标准
- 源码在哪
- 头文件在哪
- 需要链接哪些目标

当前模板的 `CMakeLists.txt` 有一个非常关键的特点：

- 它会自动扫描 `User/` 目录下的 `.c/.cpp`
- 它也会自动收集 `User/` 下头文件所在目录

所以在当前模板里：

- **如果你新增的源码文件放在 `User/` 目录下，不需要再手动修改 `CMakeLists.txt`**

原因是当前 `CMakeLists.txt` 已经通过 `GLOB_RECURSE` 把 `User/` 目录下的源码和头文件路径都自动收进来了。

但你仍然要注意：

- 如果你把代码放到了 `User/` 之外，就可能需要手动补路径
- 如果 VS Code 没刷新，要重新 `Configure` 或 `Reconfigure CMake`

### `startup_stm32f427xx.s`

启动文件。单片机上电后最先执行的就是这里，主要负责栈初始化、中断向量表配置，然后跳转到 `main()`。

### `STM32F427XX_FLASH.ld`

链接脚本。负责告诉链接器 Flash 和 RAM 如何分配。

### `test_feedback.ioc`

CubeMX 工程文件。时钟、CAN、UART、TIM、GPIO 等配置都保存在这里。

## 5. `User/` 总体结构

`User/` 是整个模板最重要的目录，结构如下：

- `Algorithm/`：算法
- `Device/`：设备对象
- `Driver/`：通信和底层接口封装
- `Module/`：功能模块
- `Protocol/`：协议
- `Task/`：FreeRTOS 任务和回调分发

后面按目录分别详细说明。

阅读代码时建议这样做：

1. **先看 `.h` 头文件**
2. **先看类定义、函数声明、结构体、枚举、成员变量**
3. **先搞清楚“这个模块对外提供什么接口”**
4. 再去看 `.cpp` 里的具体实现

原因是：

- **头文件决定“这个模块有什么”**
- **源文件决定“这个模块怎么实现”**

所以读 `cpp` 前，**先把对应 `h` 文件看一遍**，效率会高很多。

## 6. `User/Algorithm/` 算法层详解

算法层只负责计算，不直接关心具体硬件是谁。

### `User/Algorithm/Filter/alg_filter.h`

这个头文件定义了滤波器类，当前主要包括：

- 傅里叶滤波器模板类
- 卡尔曼滤波器类

常见调用步骤是：

1. 先 `Init()` 初始化滤波器参数
2. 每次拿到新数据后调用 `Set_Now()`
3. 周期调用 `TIM_Calculate_PeriodElapsedCallback()`
4. 用 `Get_Out()` 读取滤波结果

这类文件常用于：

- 陀螺仪数据滤波
- 电机速度滤波
- 传感器噪声抑制

### `User/Algorithm/Filter/alg_filter.cpp`

这里是滤波算法的实现，真正的滤波计算逻辑在这里执行。使用时一般只需要面向 `.h` 中定义的类接口调用，不需要直接碰内部实现。

### `User/Algorithm/PID/alg_pid.h`

这是工程里很重要的一个文件，定义了 `Class_PID`。

#### `Class_PID` 常用函数

`Init(...)`

- 用来初始化 PID 参数
- 一般在任务初始化或者模块初始化阶段调用一次
- 常见参数包括 `Kp`、`Ki`、`Kd`、积分限幅、输出限幅、计算周期等

**`Init(...)` 里初始化的主要内容是：**

- **比例系数 `K_P`**
- **积分系数 `K_I`**
- **微分系数 `K_D`**
- **前馈系数 `K_F`**
- **积分限幅 `I_Out_Max`**
- **输出限幅 `Out_Max`**
- **计算周期 `D_T`**
- **死区 `Dead_Zone`**
- **变速积分、积分分离相关参数**

这些参数可以简单理解成：

- **`K_P`：误差有多大，先给多大修正**
- **`K_I`：长期误差要不要慢慢补**
- **`K_D`：变化太快时要不要提前压一下**
- **`Out_Max`：最终输出最大允许多大**
- **`D_T`：默认多久算一次**

`Set_Target(float)`

- 设置目标值
- 例如目标速度、目标角度、目标电流

`Set_Now(float)`

- 设置当前反馈值
- 例如当前速度、当前角度

`TIM_Adjust_PeriodElapsedCallback()`

- 执行一次 PID 计算
- 一般在周期任务里调用，例如 1ms 调一次

`Get_Out()`

- 读取本次 PID 的输出值

#### 最基本的 PID 调用顺序

```cpp
pid.Init(...);

for (;;)
{
    pid.Set_Target(target);
    pid.Set_Now(now);
    pid.TIM_Adjust_PeriodElapsedCallback();
    float out = pid.Get_Out();
}
```

### `User/Algorithm/PID/alg_pid.cpp`

这里实现了：

- P、I、D 计算
- 积分限幅
- 输出限幅
- 死区
- 变速积分
- 积分分离
- 前馈

调电机时如果发现“响应慢、超调大、一直抖”，通常最终都要回到这里对应的参数使用方式上理解。

### `User/Algorithm/Scope/alg_slope.h`

这个文件定义了 `Class_Slope`，用于把目标值平滑变化，避免目标一步跳太大。

常见调用顺序：

1. `Init()` 初始化增减速率
2. `Set_Target()` 设置目标值
3. `Set_Now_Real()` 输入当前实际值
4. 周期调用 `TIM_Calculate_PeriodElapsedCallback()`
5. `Get_Out()` 读取平滑后的目标

**`Init()` 里初始化的主要内容是：**

- **加速步长 `Increase_Value`**
- **减速步长 `Decrease_Value`**
- **规划优先级 `Slope_First`**

它们的含义是：

- **加速步长**：每个控制周期最多往上增加多少
- **减速步长**：每个控制周期最多往下减少多少
- **`Slope_First_REAL` / `Slope_First_TARGET`**：规划时优先参考真实值还是目标值

适合用在：

- 底盘速度缓启动
- 云台角度平滑过渡

### `User/Algorithm/Scope/alg_slope.cpp`

这里实现斜坡规划算法。你平时主要是调用接口，不直接改这里。

### 7.2 `User/Driver/CAN/`

这一部分是整个电机链路里最关键的底层驱动。

### `drv_can.h`

这个头文件定义了：

- `Struct_CAN_Rx_Buffer`
- `Struct_CAN_Manage_Object`
- `CAN_Call_Back`
- `CAN_Init()`
- `CAN_Filter_Mask_Config()`
- `CAN_Send_Data()`
- `TIM_CAN_PeriodElapsedCallback()`

#### `Struct_CAN_Rx_Buffer`

作用：

- 保存一帧收到的 CAN 数据
- 里面有 `Header` 和 `Data[8]`

#### `CAN_Call_Back`

作用：

- 定义“收到 CAN 帧后，上层要执行什么函数”
- 也就是说，`CAN_Init()` 不是只开 CAN，它还要注册一个接收回调函数

### `drv_can.c`

#### `CAN_Init(CAN_HandleTypeDef *hcan, CAN_Call_Back Callback_Function)`

这是 CAN 的总初始化入口。

它做了几件事：

1. 调用 `HAL_CAN_Start(hcan)` 启动 CAN
2. 打开 FIFO0 和 FIFO1 的接收中断
3. 把传进来的回调函数保存起来
4. 给当前 CAN 配置过滤器

**`CAN_Init(...)` 初始化的主要内容是：**

- **启动指定的 CAN 外设**
- **打开接收中断**
- **登记接收回调函数**
- **配置默认过滤器**
- **把这一路 CAN 和它对应的管理对象绑定起来**

#### 当前模板里怎么调用

当前模板在 [tsk_config_and_callback.cpp](F:/Code/ROBOCON/2026ROBOCON/Task/BSP+FreeRTOS/Template_Cmake/User/Task/tsk_config_and_callback/tsk_config_and_callback.cpp) 里这样调用：

```cpp
CAN_Init(&hcan1, CAN_Motor_Call_Back);
```

这句话的意思是：

- 初始化 `CAN1`
- 注册一个接收回调函数 `CAN_Motor_Call_Back`

#### 回调函数要干什么

回调函数的职责不是做复杂控制，而是：

- 判断收到的是哪一个设备的报文
- 按 `StdId` 分发给对应设备对象

当前模板里的做法是：

```cpp
static void CAN_Motor_Call_Back(Struct_CAN_Rx_Buffer *Rx_Buffer)
{
    switch (Rx_Buffer->Header.StdId)
    {
        case 0x201:
            DJI_M3508_CAN_RxCpltCallback(Rx_Buffer->Data);
            break;
        case 0x00:
            J4310_CAN_RxCpltCallback(Rx_Buffer->Data);
            break;
        default:
            break;
    }
}
```

也就是说：

- `CAN_Init()` 负责启动总线并登记回调
- `CAN_Motor_Call_Back()` 负责根据 `StdId` 把数据转发给具体电机

#### `CAN_Filter_Mask_Config(...)`

作用：

- 配置 CAN 过滤器
- 决定哪些帧会被当前 CAN 接收

在这个模板里，`CAN_Init()` 内部已经帮你调用了，所以多数情况下你不需要手动再配一遍，除非你后续要改过滤策略。

#### `CAN_Send_Data(...)`

作用：

- 发送一帧 CAN 数据

参数含义：

- `hcan`：使用哪一路 CAN
- `ID`：发送的标准帧 ID
- `Data`：发送的数据
- `Length`：数据长度

#### `TIM_CAN_PeriodElapsedCallback()`

**这是 CAN 的发送部分，不是接收部分。**

它的作用不是解析数据，而是：

- 把已经准备好的发送缓存发出去

当前模板中，它会发送：

- `0x200` 对应的数据区
- `0x3fe` 对应的数据区

**当前示例代码里已经写好的发送对象是 `CAN1`。**

如果你后续还需要发送 `CAN2`，需要在这个函数里**手动补充 `CAN_Send_Data(&hcan2, ...)` 相关代码**。

也就是说，这个函数当前不是“自动把所有 CAN 都发一遍”，而是：

- **示例里已经写好的发 `CAN1`**
- **要不要发 `CAN2`，需要你自己加**

所以这里请直接记住：

- **接收：靠 CAN 中断回调**
- **发送：靠 `TIM_CAN_PeriodElapsedCallback()` 周期下发**

这意味着：

- 电机设备对象先把控制量写进发送缓存
- 然后周期调用 `TIM_CAN_PeriodElapsedCallback()`
- 最后 CAN 驱动统一把缓存真正发出去

#### HAL 的两个接收回调

`HAL_CAN_RxFifo0MsgPendingCallback()` 和 `HAL_CAN_RxFifo1MsgPendingCallback()`

它们的流程是：

1. 从硬件 FIFO 取出一帧数据
2. 放进 `CANx_Manage_Object.Rx_Buffer`
3. 调用之前注册的 `Callback_Function`

所以真正的接收链路是：

**CAN 中断 -> HAL 回调 -> `CAN1_Manage_Object.Callback_Function` -> 你的业务分发函数**

### 7.5 `User/Driver/UART/`

### `drv_uart.h`

这里定义了 UART 管理对象和常用接口：

- `UART_Init()`
- `UART_Send_Data()`
- `TIM_UART_PeriodElapsedCallback()`

用法和 CAN 很像：

- 先初始化串口
- 注册接收回调
- 周期处理或者直接发送

### `drv_uart.c`

UART 的实际收发和 HAL 回调都在这里实现。

#### `UART_Init(UART_HandleTypeDef *huart, UART_Call_Back Callback_Function, uint16_t Rx_Buffer_Length)`

作用：

1. 绑定对应串口的管理对象
2. 保存你传进来的回调函数
3. 启动 `HAL_UARTEx_ReceiveToIdle_DMA(...)`

**`UART_Init(...)` 初始化的主要内容是：**

- **选择使用哪一路 UART**
- **把 UART 和对应管理对象绑定**
- **登记接收回调函数**
- **设置接收缓冲区长度**
- **启动 DMA + 空闲中断接收**

它和 `CAN_Init()` 很像，核心也是两件事：

- 初始化外设
- 注册一个“收到数据以后该调用谁”的回调函数

例如：

```cpp
UART_Init(&huart2, UART_Serialplot_Call_Back, 64);
```

这句话的意思是：

- 初始化 `UART2`
- 接收缓冲区长度设为 `64`
- 当 `UART2` 收到一帧数据后，交给 `UART_Serialplot_Call_Back`

#### UART 回调函数应该做什么

UART 回调函数和 CAN 回调函数思路一样，职责也是：

- 判断这一帧数据该交给谁
- 再转发给对应对象处理

例如，如果你要把串口收到的数据交给串口绘图对象，可以写成：

```cpp
static Class_Serialplot serialplot;

static void UART_Serialplot_Call_Back(uint8_t *Buffer, uint16_t Length)
{
    (void)Length;
    serialplot.UART_RxCpltCallback(Buffer);
}
```

这样链路就是：

```text
UART_Init(&huart2, UART_Serialplot_Call_Back, 64)
  -> HAL_UARTEx_RxEventCallback()
  -> UART2_Manage_Object.Callback_Function(Buffer, Size)
  -> UART_Serialplot_Call_Back(Buffer, Size)
  -> serialplot.UART_RxCpltCallback(Buffer)
```

也就是说：

- `UART_Init()` 负责初始化 UART 并登记回调
- `HAL_UARTEx_RxEventCallback()` 负责把收到的数据转发给登记好的回调
- 你的业务回调负责把数据再转给具体设备对象

## 8. `User/Device/` 设备层详解

设备层负责把原始通信数据包装成“可直接操作的设备对象”。

---

### 8.1 `User/Device/Motor/Motor_DJI/`

这是大疆电机设备对象的核心目录。

### `dvc_motor.h`

这个头文件定义了：

- 电机状态枚举
- 电机 ID 枚举
- 控制方式枚举
- `Class_Motor_GM6020`
- `Class_Motor_C610`
- `Class_Motor_C620`

当前模板在 M3508 场景里主要用的是 `Class_Motor_C620`。

#### `Enum_Control_Method`

当前支持的控制方式有：

- `Control_Method_OPENLOOP`：开环
- `Control_Method_TORQUE`：扭矩控制
- `Control_Method_OMEGA`：速度控制
- `Control_Method_ANGLE`：角度控制

你初始化电机对象时，必须明确告诉它当前要用哪种控制方式。

#### `Class_Motor_C620::Init(...)`

作用：

- 把电机对象和某一路 CAN 绑定
- 指定该电机的 CAN ID
- 指定控制模式
- 设置减速比、最大扭矩等参数

**`Init(...)` 初始化的主要内容是：**

- **电机使用哪一路 CAN 总线**
- **电机的 CAN ID**
- **控制模式**
- **减速比**
- **最大扭矩**
- **内部发送缓存映射**

典型调用：

```cpp
motor_3508.Init(&hcan1, CAN_Motor_ID_0x201, Control_Method_OMEGA);
```

这句话的意思是：

- 这台电机走 `CAN1`
- 电机 ID 是 `0x201`
- 采用速度控制模式

#### `Set_Target_Omega()` / `Set_Target_Angle()` / `Set_Target_Torque()`

作用：

- 给电机设置目标值

你用哪一个取决于当前控制模式：

- 速度模式常用 `Set_Target_Omega()`
- 角度模式常用 `Set_Target_Angle()`
- 扭矩模式常用 `Set_Target_Torque()`

#### `CAN_RxCpltCallback(uint8_t *Rx_Data)`

作用：

- 接收一帧电机反馈数据
- 解析编码器、速度、扭矩、温度
- 更新 `Now_Angle`、`Now_Omega`、`Now_Torque` 等状态

注意：

- 这个函数不是你手动在循环里调的
- 它通常由 CAN 接收回调分发过来

#### `TIM_PID_PeriodElapsedCallback()`

这是电机控制里最重要的周期函数之一。

作用：

1. 根据控制模式决定走哪一层控制
2. 调用 PID 算法
3. 计算出输出量 `Out`
4. 把输出量写进对应的 CAN 发送缓存

例如在 `Control_Method_OMEGA` 模式下，它会：

1. 用 `Target_Omega` 和 `Now_Omega` 算速度环
2. 得到目标扭矩
3. 再把结果写入发送缓存

#### `TIM_Alive_PeriodElapsedCallback()`

作用：

- 判断这段时间内有没有持续收到反馈
- 如果没有，就把电机状态标为离线
- 同时清空 PID 积分，避免离线后继续累积错误

### `dvc_motor.cpp`

这个文件完成了上面所有函数的具体实现。

你最需要知道的是它内部做了两件事：

1. `Init()` 时，会根据 `hcan` 和电机 ID 找到对应的发送缓存区
2. `TIM_PID_PeriodElapsedCallback()` 时，会把控制结果写入那块缓存

也就是说，**设备对象本身不直接发 CAN 帧，它只是把“待发送数据”准备好，真正发送由 CAN 驱动层完成。**

#### 常用函数

`Init()`

- 初始化底盘内的电机对象和 PID

- 每 1ms 执行一次底盘控制流程

- 把某个 CAN 报文转给底盘内部对应的电机对象

### `User/Module/Chassis/crt_wheelchassis.cpp`

实现内容主要包括：

- 电机 PID 初始化
- 运动学逆解
- 从轮速反推底盘速度
- 把底盘目标转换为单个电机目标

这部分更适合看作“底盘基础控制内核”，而不是完整的上位机控制应用。

### 9.2 `User/Module/Chassis/uart_wheelchassis/`

这是在 `crt_wheelchassis` 基础上进一步封装的一个应用示例。

和 `crt_wheelchassis` 相比，它新增了两类更接近实际应用的逻辑：

- 串口/上位机输入目标速度
- 斜坡规划器对速度目标做平滑处理

#### `uart_wheelchassis.h`

这里定义了 `Class_Uart_Wheelchassis`，它继承自 `Class_Chassis_Omni`。

常用函数包括：

`Init(CAN_HandleTypeDef* hcan)`

- 先调用 `Class_Chassis_Omni::Init()` 完成底盘基础初始化
- 再按传入的 `hcan` 重新绑定 4 个底盘电机
- 最后初始化斜坡规划器

**这个 `Init(...)` 里初始化的主要内容是：**

- **复用底层底盘类的基础初始化**
- **重新绑定 4 个底盘电机使用的 CAN 总线**
- **初始化 `slope_vx_ / slope_vy_ / slope_vw_` 三个斜坡规划器**
- **把初始目标速度清零**

`SetMotion(float vx, float vy, float vw)`

- 设置上位机给定的原始目标速度
- 同时会做死区处理，避免很小的输入导致抖动

`WithSlopeTask(float vx, float vy, float vw)`

- 先更新目标速度
- 再执行一次斜坡规划
- 然后把规划后的速度写到底盘目标中
- 最后调用底盘 1ms 控制周期

`WithSlopeTask()`

- 使用已经缓存好的目标速度直接做一次斜坡规划和底盘控制

`SetSlopeStep(...)`

- 动态调整斜坡规划器的增减速步长

#### `uart_wheelchassis.cpp`

这个文件是一个很好的示例，因为它把“基础底盘类”和“应用层速度规划”区分开了。

它当前的实际流程是：

1. 外部通过 `SetMotion()` 或 `WithSlopeTask(vx, vy, vw)` 给目标速度
2. `Class_Slope` 规划器对 `vx / vy / vw` 做平滑过渡
3. 再调用 `Set_Target_Velocity_X/Y/Omega()` 把规划后的结果交给底盘基础类
4. 最后调用 `TIM_1ms_Control_PeriodElapsedCallback()` 完成底盘闭环控制

所以你可以把两者关系理解为：

- `crt_wheelchassis`：底盘基础控制框架
- `uart_wheelchassis`：基于底盘基础控制框架的一个串口控制应用示例

### 11.1 `User/Task/tsk_config_and_callback/`

这是整个工程的关键目录之一，主要用来存放：

- 全局初始化函数
- 全局回调函数

它不负责某一台电机或某一个设备的具体业务循环，而是负责“系统级入口”。

### `tsk_config_and_callback.h`

当前只对外暴露：

```c
void Task_Init(void);
```

### `tsk_config_and_callback.cpp`

这个文件当前主要有两个核心部分。

#### `static void CAN_Motor_Call_Back(Struct_CAN_Rx_Buffer *Rx_Buffer)`

作用：

- 这是注册给 `CAN_Init()` 的回调函数
- 每收到一帧 CAN 数据，它就会被调用
- 它通过 `Rx_Buffer->Header.StdId` 判断当前报文属于哪个设备
- 然后把 `Rx_Buffer->Data` 交给对应设备对象的回调函数

你以后新增一个 CAN 设备时，通常要在这里补一条分发逻辑。

#### `void Task_Init(void)`

作用：

- 在 FreeRTOS 创建线程前，统一完成全局初始化
- 这个函数不是随便找地方调用的，它被放在 `freertos.c` 的 `MX_FREERTOS_Init()` 里面执行

在 [freertos.c](F:/Code/ROBOCON/2026ROBOCON/Task/BSP+FreeRTOS/Template_Cmake/Core/Src/freertos.c) 中，当前实际关系是：

```c
void MX_FREERTOS_Init(void) {
  Task_Init();

  TaskJ4310Handle = osThreadNew(StartTaskJ4310, NULL, &TaskJ4310_attributes);
  Task3508Handle = osThreadNew(StartTask3508, NULL, &Task3508_attributes);
}
```

也就是说：

- `Task_Init()` 先执行
- 线程后创建

所以 `tsk_config_and_callback.cpp` 的定位要明确：

- 放系统级初始化函数
- 放系统级回调函数
- 不放某个任务自己的主循环

当前模板里它做了这些事情：

1. `BSP_Init(...)`：打开板级电源
2. `CAN_Init(&hcan1, CAN_Motor_Call_Back)`：初始化 CAN1 并注册回调
3. `DJI_M3508_Task_Init()`：初始化大疆电机任务相关资源
4. `J4310_Task_Init()`：初始化达妙电机任务相关资源

也可以把它理解成初始化了三大块内容：

- **板级资源**
- **全局通信回调**
- **示例任务所需的设备对象**

这个顺序非常重要：

- **先把板级资源准备好**
- **再把 CAN 总线启动并挂上回调**
- **再初始化电机对象**

如果顺序反了，可能出现电机对象已经建好了，但 CAN 还没启动，或者回调还没注册。

## 12. FreeRTOS 线程是怎么启动的？

在 [freertos.c](F:/Code/ROBOCON/2026ROBOCON/Task/BSP+FreeRTOS/Template_Cmake/Core/Src/freertos.c) 中，流程是这样的：

1. `MX_FREERTOS_Init()` 被调用
2. 它先执行 `Task_Init()`
3. 然后创建各个线程，例如 `TaskJ4310`、`Task3508`
4. 线程入口函数 `StartTask3508()` 内部再调用 `DJI_M3508_Task(argument)`

所以当前模板的任务启动顺序是：

```text
MX_FREERTOS_Init()
  -> Task_Init()
  -> osThreadNew(StartTask3508, ...)
  -> StartTask3508()
  -> DJI_M3508_Task()
```

这个顺序一定要搞清楚，因为很多初始化到底该写在 `Task_Init()` 还是写在 `DJI_M3508_Task_Init()`，就取决于你理解不理解这个启动流程。

## 13. 电机从上电到闭环控制的完整链路

这一节是最重要的应用说明。

下面以当前模板中的大疆 M3508 电机为例，按真实代码顺序讲完整流程。

### 第一步：CubeMX 先把底层外设初始化好

在 `main.c` 和 CubeMX 生成的初始化代码中，会先完成：

- 时钟初始化
- GPIO 初始化
- CAN1 初始化
- UART 初始化
- FreeRTOS 初始化入口准备

这一步是 HAL 层准备，不是你自己在 `User/` 里写的业务初始化。

### 第二步：进入 `MX_FREERTOS_Init()`

在 [freertos.c](F:/Code/ROBOCON/2026ROBOCON/Task/BSP+FreeRTOS/Template_Cmake/Core/Src/freertos.c) 中，先调用：

```c
Task_Init();
```

### 第三步：`Task_Init()` 初始化板级资源和 CAN

在 [tsk_config_and_callback.cpp](F:/Code/ROBOCON/2026ROBOCON/Task/BSP+FreeRTOS/Template_Cmake/User/Task/tsk_config_and_callback/tsk_config_and_callback.cpp) 中：

```cpp
BSP_Init(...);
CAN_Init(&hcan1, CAN_Motor_Call_Back);
DJI_M3508_Task_Init();
J4310_Task_Init();
```

这里做的事是：

1. 开板子上的相关电源
2. 启动 `CAN1`
3. 注册 CAN 接收回调函数 `CAN_Motor_Call_Back`
4. 初始化各个电机任务所需的对象

### 第四步：初始化电机对象和 PID

在 [motor_dji_task.cpp](F:/Code/ROBOCON/2026ROBOCON/Task/BSP+FreeRTOS/Template_Cmake/User/Task/Motor_DJI/motor_dji_task.cpp) 中：

```cpp
motor_3508.PID_Omega.Init(450.0f, 279.936f, 0.0f, 0.0f, 10000.0f, 10000.0f);
motor_3508.Init(&hcan1, CAN_Motor_ID_0x201, Control_Method_OMEGA);
```

这里要分清楚两件事：

#### 1. 初始化 PID

`motor_3508.PID_Omega.Init(...)`

作用是：

- 给速度环设置 PID 参数
- 这一步只是在准备控制器

#### 2. 初始化电机对象

`motor_3508.Init(&hcan1, CAN_Motor_ID_0x201, Control_Method_OMEGA)`

作用是：

- 指定电机挂在哪一路 CAN 上：`CAN1`
- 指定电机的 ID：`0x201`
- 指定控制模式：速度控制

这一步是在准备“这个电机设备对象应该怎样和外部通信”。

更完整地说，这一步初始化的是：

- **电机使用哪一路 CAN**
- **电机 ID 是多少**
- **当前控制模式是什么**
- **内部使用哪块发送缓存**

### 第五步：创建线程并进入任务循环

FreeRTOS 创建 `Task3508` 后，会进入：

```cpp
DJI_M3508_Task(argument);
```

在循环里，当前模板执行的是：

```cpp
motor_3508.Set_Target_Omega(5.0f);
motor_3508.TIM_PID_PeriodElapsedCallback();
TIM_CAN_PeriodElapsedCallback();
osDelay(1);
```

含义是：

1. 设置目标速度
2. 根据目标值和反馈值做一次控制计算
3. 把计算结果通过 CAN 发出去
4. 每 1ms 跑一次

### 第六步：`TIM_PID_PeriodElapsedCallback()` 内部完成控制计算

对于 `Control_Method_OMEGA`，电机对象内部会做：

1. 读取 `Target_Omega`
2. 读取当前反馈 `Now_Omega`
3. 调用 `PID_Omega` 计算输出
4. 把输出写进对应的 CAN 发送缓存

注意：

- 到这一步为止，**只是把发送数据准备好了**
- 还没有真正把报文发到总线上

### 第七步：`TIM_CAN_PeriodElapsedCallback()` 统一发送缓存

CAN 驱动层的 `TIM_CAN_PeriodElapsedCallback()` 会把缓存发出去：

```cpp
CAN_Send_Data(&hcan1, 0x200, CAN1_0x200_Tx_Data, 8);
```

这时电机驱动器才真正收到控制命令。

这里要特别注意：

- **当前示例代码发的是 `CAN1`**
- **如果你的控制对象在 `CAN2` 上，就要自己补 `CAN_Send_Data(&hcan2, ...)`**

### 第八步：电机返回反馈报文

电机返回反馈后，硬件会触发 CAN 接收中断，流程如下：

```text
HAL_CAN_RxFifo0MsgPendingCallback()
  -> CAN1_Manage_Object.Callback_Function(...)
  -> CAN_Motor_Call_Back(...)
  -> DJI_M3508_CAN_RxCpltCallback(...)
  -> motor_3508.CAN_RxCpltCallback(...)
```

### 第九步：设备对象解析反馈

`motor_3508.CAN_RxCpltCallback(Rx_Data)` 会解析：

- 编码器
- 转速
- 扭矩
- 温度

并更新：

- `Now_Angle`
- `Now_Omega`
- `Now_Torque`

这样下一个任务周期到来时，PID 就能拿到最新反馈继续闭环控制。

### 一句话总结完整链路

```text
初始化 CAN
-> 注册 CAN 回调
-> 初始化电机对象
-> 初始化电机 PID
-> 任务循环设置目标值
-> 电机对象执行 PID 计算
-> CAN 驱动发送控制帧
-> 电机返回反馈
-> 回调分发到电机对象
-> 电机对象更新反馈状态
-> 下一周期继续闭环
```

这就是当前模板里“电机应用”的完整流程。

## 15. 场景 2：我要写一个底盘控制逻辑，应该怎么做？

建议顺序如下。

### 1. 算法层准备控制方法

通常会用到：

- `alg_pid`
- `alg_filter`
- `alg_slope`

### 2. 设备层准备执行对象

例如大疆底盘电机：

- `dvc_motor.h`
- `dvc_motor.cpp`

### 3. 模块层组织底盘逻辑

在：

- `crt_wheelchassis.h`
- `crt_wheelchassis.cpp`

里完成：

- 目标速度管理
- 运动学解算
- 各轮目标分配
- 各轮电机目标下发

### 4. 任务层周期调用

例如每 1ms：

1. 读取遥控或上位机目标
2. 设置底盘目标速度
3. 调底盘控制回调
4. 调 CAN 周期发送

重点原则：

- 底盘整体逻辑放 `Module`
- 单个电机对象放 `Device`
- 周期调度放 `Task`

不要把底盘整体逻辑直接拆散写在多个单电机任务里。

## 16. 常见坑

### 1. 不要只初始化电机对象，不初始化 CAN

如果只写了：

```cpp
motor_3508.Init(...);
```

但没有先：

```cpp
CAN_Init(&hcan1, CAN_Motor_Call_Back);
```

那电机对象虽然存在，但 CAN 总线没启动，也没有接收回调，系统不会正常闭环。

### 2. 不要只启动 CAN，不做回调分发

如果 CAN 已经初始化，但你没有在回调里按 `StdId` 分发给对应设备对象，那么现象通常是：

- 任务在跑
- 报文也在收
- 但电机反馈状态不更新

### 3. 不要忘记初始化 PID

如果电机对象初始化了，但 `PID_Omega.Init(...)` 没做，控制效果一定不正常，甚至完全不可用。

### 4. 任务里做计算，不要在中断里做重活

中断回调里适合做：

- 收数据
- 分发数据
- 更新状态

不适合做：

- 大量打印
- 复杂控制逻辑
- 长时间阻塞

### 5. 当前模板下，新增 `User/` 内源码通常不需要手动加到 CMake

因为当前 `CMakeLists.txt` 已经自动扫描 `User/` 下的所有源码和头文件目录。

所以结论可以直接记成：

- **只要文件放在 `User/` 里面，新增文件时不需要改 `CMakeLists.txt`**

但新增后仍然建议：

1. 重新 `Configure CMake`
2. 编译一次
3. 确认没有缺文件、缺头文件问题

### 6. 不要乱改 CubeMX 自动生成区

如果必须改，优先写在 `USER CODE` 区域内。
